# spec.md 템플릿

기능 성격에 맞춰 섹션을 취사하되, **가이드 필수(✅) 6개는 항상 넣는다.** 나머지는 해당될 때만.
이 필수 목록은 `references/canonical/spec-driven-development.md` §4-3 이 규정한 것이다 (canonical 원본).
`//`는 지침(문서에 남기지 않음). 모르는 값은 `[확인 필요: ...]`로.

## 섹션 선택 가이드

| 섹션 | 언제 | 필수 |
|------|------|------|
| 개요 | 항상 | ✅ 가이드 필수 |
| 기술 스택 | 항상 (이 스펙이 쓰는 라이브러리·SDK·프레임워크) | ✅ 가이드 필수 |
| 실행 환경 | 항상 (런타임·플랫폼·버전) | ✅ 가이드 필수 |
| 데이터 구조(모델) | 데이터를 다루면 (거의 항상) | ✅ 가이드 필수 |
| 파일(페이지) 구성 | 항상 (생성/수정 파일 목록 표) | ✅ 가이드 필수 |
| 변경 이력 | 항상 | ✅ 가이드 필수 |
| 라우팅 / 접근 제어 | 페이지·권한이 있는 기능 | 화면 기능이면 |
| 페이지 구성 (ASCII 와이어프레임) | UI가 있는 기능 | UI면 |
| 섹션별 상세 (기능·UI·데이터) | UI가 있는 기능 | UI면 |
| 데이터 저장 구조 (컬렉션/테이블) | DB를 쓰면 | DB면 |
| 기술 구현 (컴포넌트·서버 액션·함수) | 로직·구조가 있으면 | 권장 |
| 실행 스크립트 / 환경 변수 | 빌드·설정 값이 있으면 | 해당 시 |
| API 엔드포인트 | 서버 통신 | 통신이면 |
| 의존성 / 관련 스펙 변경 | 다른 스펙과 얽히면 | 해당 시 |

> **역할 분리 주의:** 프로젝트 전역 스택·실행 환경이 이미 프로젝트 `CLAUDE.md`(AI 컨텍스트 파일)에 있으면, spec.md에는 "프로젝트 공통 + 이 스펙 고유: 〈...〉"만 한 줄로 적어 중복을 피한다. 프로젝트 문서가 없으면 spec.md에 명시한다.

**가이드 부합 셀프 체크:** 위 ✅ 6개(개요·기술 스택·실행 환경·데이터 구조·파일 구성·변경 이력)가 모두 있는가?

## 골격

```markdown
# spec_{기능명}

> 〈이 스펙이 무엇인지 1~2줄. 누가 쓰고 무엇을 하는지.〉

## 1. 개요
// 무엇을·왜. 핵심 동작 요약.
〈개요〉

## 기술 스택        // (가이드 필수) 이 스펙이 쓰는 라이브러리·SDK·프레임워크
- 〈예: Next.js 16 App Router, Firestore, next-intl〉
// 프로젝트 CLAUDE.md에 전역 스택이 있으면 "프로젝트 공통 + 이 스펙 고유: 〈...〉"만.

## 실행 환경        // (가이드 필수) 런타임·플랫폼·버전
- 〈예: Node 20 / 브라우저(모바일·데스크톱) / Vercel 배포〉

### 라우팅            // (화면 기능이면)
경로: `/〈path〉`
\`\`\`
(app)
└── 〈folder〉/
    └── page.tsx
\`\`\`

## 2. 접근 제어        // (권한이 있으면)
〈누가 접근 가능한지, 어떻게 검증하는지 — 흐름으로〉

## 3. 페이지 구성       // (UI면 — ASCII 와이어프레임으로 배치를 보여준다)
\`\`\`
┌─────────────────────────────┐
│  〈화면 제목〉               │
│  ┌───────────────────────┐  │
│  │ 〈섹션/카드〉          │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
\`\`\`

## 4. 섹션별 상세       // (UI면 — 각 섹션의 기능/UI/데이터)
### 섹션 1 — 〈이름〉
- **기능**: 〈무엇을 하는지〉
- **UI**: 〈레이블·컨트롤·상태·빈 상태 문구〉
- **데이터**: 〈읽고 쓰는 필드〉

## 5. 데이터 모델       // (필수 — 실제 타입으로)
\`\`\`typescript
interface 〈Name〉 {
  〈field〉: 〈type〉;   // 〈설명〉
}
type 〈Enum〉 = '〈a〉' | '〈b〉';
\`\`\`
// 초기값·기본값 규칙이 있으면 표로 명시 (문서 없을 때 어떻게 동작하는지 등).

## 6. 데이터 저장 구조   // (DB면 — 컬렉션/테이블 구조)
\`\`\`
〈collection〉/
  〈doc/row〉   ← 〈설명〉
\`\`\`

## 7. 기술 구현        // (필수)
### 컴포넌트 구조
\`\`\`
〈path〉/
  page.tsx                  ← 〈역할〉
  _components/
    〈Component〉.tsx        ← 〈역할〉
\`\`\`
### 서버 액션 / 함수
\`\`\`
〈file〉
  ├── 〈fn〉()   ← 〈역할〉
\`\`\`

## 8. API 엔드포인트     // (서버 통신이면)
| 메서드 | 경로 | 설명 |
| --- | --- | --- |
| 〈GET〉 | 〈/api/...〉 | 〈...〉 |

## 9. 의존성           // (있으면)
- `spec_〈other〉`의 〈타입/기능〉이 확정돼야 〈이것〉 구현 가능

## 10. 관련 스펙 변경    // (다른 스펙을 건드리면)
| 스펙 | 변경 내용 |
| --- | --- |
| `spec_〈x〉` | 〈...〉 |

## 파일(페이지) 구성   // (가이드 필수) 이 기능에서 생성/수정하는 파일 목록
| 파일 | 경로 | 설명 |
| --- | --- | --- |
| 〈file.tsx〉 | 〈src/app/...〉 | 〈역할〉 |

## 11. 변경 이력        // (필수)
| 날짜 | 변경 내용 |
| --- | --- |
| 〈YYYY-MM-DD〉 | 최초 작성 |
```

## 예시 발췌 (spec_admin — 실제 스펙, 축약)

리치 스펙이 어느 정도 상세한지 감을 잡는 기준. 화면 기능 스펙의 모범.

```markdown
# spec_admin
> 앱 전반의 설정을 관리하는 어드민 페이지 스펙. `admin` 권한 보유 멤버만 접근.

## 2. 접근 제어
Zustand 역할은 로그인 시점 스냅샷이므로, 진입 시 항상 권한 서비스에서 admin 권한을 재확인한다.
  사용자가 /admin 진입 → checkToolPermission(email,'admin')
    allowed:true  → 렌더 / allowed:false → /dashboard redirect
- 재확인은 Server Action으로 서버 사이드 수행. 확인 중 로딩 스피너.

## 4. 섹션 1 — 앱 설정
### 기능: 슬랙 알림 전송 여부
| 상태 | 동작 |
| ON | 요청 제출 시 지정 채널로 메시지 전송 |
| OFF | 전송 없음 |
- 토글 변경 즉시 Firestore 저장(별도 버튼 없음). 저장 중 토글 비활성화.
- Firestore `app_settings/global`의 `slackNotificationEnabled` 필드.

## 7. 데이터 모델
interface AppSettings {
  slackNotificationEnabled: boolean;
  slackChannels: SlackChannel[];
  changeHistory: AdminChangeLog[];   // 현재 페이지 이력
  historyPage: number;
}
#### 초기값 (문서가 없을 때) — Firestore를 쓰지 않고 메모리 기본값 사용
| 필드 | 초기값 | 이유 |
| slackNotificationEnabled | true | 기본 활성이어야 요청 누락 안 됨 |
```

> 관찰: 상태표·흐름도·초기값 이유·필드 단위 설명까지 구체적이다. "잘 처리한다"가 아니라 "무엇이 어떻게" 수준으로 쓴다.
