# 스펙 주도 개발 — 방법론

> 기능을 구현하기 전에 스펙 문서를 먼저 쓰고, 그 문서를 기준으로 AI 코딩 도구와 함께 개발한다.
> 설계 의도·데이터 구조·파일 구성을 구조화된 문서로 정리한 뒤 코드를 짠다.

## 왜 하나
- **AI 맥락 유지**: 세션이 바뀌어도 spec.md를 참조하면 목적·구조·방향이 일관된다.
- **팀 의도 공유**: 여러 명이 나눠 개발해도 설계를 따로 설명할 필요가 없다. 신규 합류자는 `.specs/`만 읽으면 된다.
- **결정 이유 보존**: "왜 이렇게 구현했지?"의 답이 PROGRESS.md 결정 기록에 남는다.
- **살아있는 설계서**: 설계가 바뀌면 spec.md도 함께 갱신 — 코드와 문서가 같은 repo에서 동기화된다.

## 파일 역할 분리 (단일 출처 원칙)

| 정보 | 작성 위치 | 여기엔 쓰지 않음 |
|------|-----------|------------------|
| 기능의 설계 (무엇을·어떤 구조로) | `spec.md` | PROGRESS.md |
| 기능의 진행 (어디까지·무엇이 남음) | `PROGRESS.md` | spec.md |
| 개발 중 기술 결정과 사유 | `PROGRESS.md` 결정 기록 | spec.md |
| 스펙 문서 자체의 수정 내역 | `spec.md` 변경 이력 | PROGRESS.md |
| 전체 스펙 목록·상태 | `INDEX.md` | 개별 spec.md |

**겹침 방지 3원칙**
1. 설계 정보는 spec.md에만 존재한다. 다른 곳에선 spec.md를 참조하게 한다.
2. 진행 정보는 PROGRESS.md에만 존재한다.
3. 전체 현황은 INDEX.md에만 존재한다.

## 워크플로우

| 순서 | 단계 | 갱신 파일 |
|------|------|-----------|
| 1 | 스펙 생성 (폴더 + spec.md/PROGRESS.md 스텁) | spec.md, INDEX.md |
| 2 | 설계 작성 (데이터 구조·파일 구성·화면) | spec.md |
| 3 | 구현 시작 (spec.md를 맥락으로) | PROGRESS.md |
| 4 | 진행 갱신 (작업 완료마다) | PROGRESS.md |
| 5 | 설계 변경 시 spec.md 먼저 갱신 후 코드 반영 | spec.md |
| 6 | 완료 시 INDEX 상태 변경 | INDEX.md, PROGRESS.md |

## 범위 잡기
- **1 스펙 = 1 기능 브랜치 = 1 PR** 단위.
- 스펙 폴더 ↔ Git 브랜치 1:1 매핑이 이상적 (`spec_user_auth` ↔ `sp5/feat_user_auth`).

## 네이밍
- `spec_{기능명}` — 영문 소문자 + 언더스코어. 목적이 드러나게.
- ✅ `spec_user_auth`, `spec_admin`, `spec_slack_messages`
- ❌ `spec-user-auth`(하이픈), `spec_UserAuth`(대문자), `user_auth`(접두사 누락), `spec_new_feature`(불명확)

## 의존성
- 다른 스펙에 의존하면 INDEX.md 의존성 칸 + 해당 spec.md에 명시한다.
  ```
  ## 의존성
  - `spec_form_schema`의 FormField 타입이 확정돼야 관리 화면 구현 가능
  ```

## 주의
- **민감 정보 금지**: API 키·토큰·서버 주소·매출/유저 지표를 스펙에 넣지 않는다 (Git 커밋 + AI 전달됨).
- **완료 스펙 삭제 금지**: 상태만 ✅로. 유지보수 시 참고 자산.
- **코드와 함께 커밋**: 스펙 변경은 관련 코드와 같은 커밋/PR에.
- **AI 자동 갱신 확인**: PROGRESS.md를 AI가 갱신하면, 안 한 걸 완료로 적거나 사유를 부정확히 남길 수 있으니 검토한다.
