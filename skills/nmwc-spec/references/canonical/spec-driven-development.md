# 스펙 주도 개발 - 공통 (범용)

> AI 코딩 도구와 함께 체계적으로 개발을 진행하기 위한
스펙 주도 개발(Spec-Driven Development) 공통 가이드입니다.
이 문서는 특정 프레임워크나 언어에 종속되지 않는 범용 버전입니다.
> 

---

## 1. 스펙 주도 개발이란?

스펙 주도 개발(Spec-Driven Development)은 기능을 구현하기 전에 **스펙 문서를 먼저 작성**하고, 이 문서를 기준으로 AI 코딩 도구와 함께 개발을 진행하는 방법론입니다.

기존 개발 방식에서는 머릿속의 설계를 바로 코드로 옮기거나, 간단한 메모를 남기고 구현을 시작합니다. 스펙 주도 개발에서는 설계 의도, 데이터 구조, 파일 구성 등을 구조화된 문서로 먼저 정리한 뒤, AI가 이 문서를 맥락으로 참고하면서 코드를 작성합니다.

### 기존 개발 방식과의 차이

| 구분 | 기존 방식 | 스펙 주도 개발 |
| --- | --- | --- |
| 설계 | 머릿속 또는 간단한 메모 | 구조화된 스펙 문서 |
| AI 활용 | 매번 맥락을 설명 | 스펙 문서를 맥락으로 전달 |
| 진행 추적 | 별도 관리 (Jira, Notion 등) | 스펙 폴더 내 PROGRESS.md |
| 기능 완료 후 | 코드만 남음 | 스펙 문서 + 결정 기록이 함께 남음 |
| 팀원 공유 | 구두 설명 또는 별도 문서 | 스펙 폴더 자체가 설계 문서 |

## 2. 왜 스펙 주도 개발을 하는가?

### 2-1. AI가 맥락을 유지하도록 돕는다

AI 코딩 도구는 대화가 길어지거나 새 세션을 시작하면 이전 맥락을 잃습니다. 스펙 문서를 항상 참조할 수 있도록 등록해두면, AI가 프로젝트의 목적, 데이터 구조, 구현 방향을 일관되게 유지할 수 있습니다.

### 2-2. 팀원 간 의도를 공유한다

스펙 문서가 있으면 같은 기능을 여러 명이 나눠서 개발할 때 설계 의도를 별도로 설명할 필요가 없습니다. 신규 팀원이 합류해도 스펙 폴더만 읽으면 현재 상태와 방향을 파악할 수 있습니다.

### 2-3. 설계 결정의 이유가 기록된다

스펙 문서의 결정 기록에 당시 논의 내용과 선택 이유를 남겨두면, 나중에 해당 기능을 수정하거나 확장할 때 과거 맥락을 빠르게 파악할 수 있습니다.

### 2-4. 문서가 곧 살아있는 설계서가 된다

스펙 문서는 한 번 쓰고 방치하는 문서가 아닙니다. 개발이 진행되면서 데이터 구조가 변경되거나 새로운 결정이 내려지면 스펙 문서도 함께 갱신됩니다. 코드와 문서가 같은 저장소에서 함께 관리되므로, 문서가 코드와 동기화된 상태를 유지합니다.

## 3. 폴더 구조

### 3-1. 기본 구조

스펙 문서와 AI 설정 파일은 **프로젝트 루트** 하위에 위치합니다. AI 도구가 프로젝트 루트를 기준으로 파일을 인식하기 때문입니다.

```
프로젝트 루트/
├── {AI 컨텍스트 파일}                # AI 도구별 진입점 (도구에 따라 다름)
├── {AI 설정 폴더}                    # AI 도구별 설정 폴더 (도구에 따라 다름)
├── .specs/                          # 스펙 문서 루트
│   ├── INDEX.md                     # 전체 스펙 목록 및 상태 총괄
│   ├── spec_user_auth/              # 개별 스펙 폴더
│   │   ├── spec.md                  # 기능 설계 문서
│   │   ├── PROGRESS.md              # 진행 상황 추적
│   │   └── references/              # 참고 자료 (선택)
│   └── spec_dashboard/
│       ├── spec.md
│       └── PROGRESS.md
└── src/                             # 소스 코드 (프로젝트마다 다름)
    └── ...
```

### 3-2. AI 컨텍스트 파일

AI 도구가 프로젝트에 진입할 때 가장 먼저 읽는 파일입니다. 도구마다 이름과 위치가 다릅니다.

| AI 도구 | 파일명 | 위치 |
| --- | --- | --- |
| Claude Code | `CLAUDE.md` | 프로젝트 루트 |
| Codex CLI | `AGENTS.md` | 프로젝트 루트 |
| Cursor | `.cursor/rules/*.mdc` | `.cursor/rules/` |
| OpenAI (ChatGPT/GPTs) | 시스템 프롬프트 또는 업로드 파일 | AI 설정 내 |

### 3-3. 스펙 폴더 경로 표기

문서와 대화에서 `.specs/` 경로를 표기할 때는 프로젝트 루트 기준의 상대 경로를 사용합니다.

```
전체 경로: /home/user/my-project/.specs/spec_user_auth/spec.md
표기 경로: .specs/spec_user_auth/spec.md
```

### 3-4. 개별 스펙 폴더 네이밍

스펙 폴더명은 `spec_{기능명}` 형식으로 작성합니다.

- 영문 소문자와 언더스코어(`_`)만 사용합니다.
- 기능의 목적을 간결하게 표현합니다.

```
✅ 좋은 예시
spec_user_auth
spec_dashboard
spec_payment_flow
spec_notification_system

❌ 나쁜 예시
spec-user-auth          # 하이픈 사용
spec_UserAuth           # 대문자 사용
user_auth               # spec_ 접두사 누락
spec_new_feature        # 목적이 불명확
```

## 4. 핵심 파일 역할과 작성법

### 4-1. AI 컨텍스트 파일

AI가 프로젝트에 진입할 때 가장 먼저 읽는 파일입니다. 프로젝트 전체의 맥락을 빠르게 전달하는 역할을 합니다.

**포함 항목**

| 항목 | 설명 |
| --- | --- |
| 프로젝트 한줄요약 | 이 프로젝트가 무엇인지 한 문장으로 설명 |
| 스펙 문서 위치 | `.specs/` 폴더 경로 및 INDEX.md 안내 |
| 빠른 참조 | 기술 스택, 빌드/실행 명령어, 환경 정보 |
| 핵심 구현 포인트 | 프로젝트 전반에 걸친 중요한 기술적 결정이나 주의사항 |

**작성 예시**

```markdown
# My Project

> React + TypeScript 기반 관리자 대시보드 웹앱.

## 스펙 문서
모든 기능 스펙은 `.specs/` 폴더에서 관리합니다.
전체 스펙 목록 및 상태는 `.specs/INDEX.md`를 참고하세요.

## 빠른 참조
- 기술 스택: React 18, TypeScript, Next.js 14
- 패키지 매니저: pnpm
- 실행: pnpm dev
- 빌드: pnpm build

## 핵심 구현 포인트
- 코딩 규약: 프로젝트 코딩 규약 참고
- 상태 관리: Zustand 사용
- API 통신: React Query + Axios
```

> **주의:** 이 파일에는 개발 이력이나 진행 상황을 작성하지 않습니다. 개발 이력은 각 스펙의 PROGRESS.md에서 관리합니다.
> 

### 4-2. INDEX.md — 전체 스펙 총괄

`.specs/INDEX.md`는 현재 프로젝트에서 관리 중인 모든 스펙의 목록과 상태를 보여주는 파일입니다.

**작성 예시**

```markdown
# 스펙 인덱스

> 전체 스펙 목록 및 상태를 관리합니다.

## 스펙 목록

| 스펙 | 상태 | 스프린트 | 담당자 | 의존성 | 최종 갱신 |
|------|------|---------|--------|--------|----------|
| [spec_user_auth](./spec_user_auth/spec.md) | 🟢 진행중 | sprint5 | 홍길동 | - | 2026-05-20 |
| [spec_dashboard](./spec_dashboard/spec.md) | 🟢 진행중 | sprint5 | 김철수 | spec_user_auth | 2026-05-19 |
| [spec_notification](./spec_notification/spec.md) | ✅ 완료 | sprint4 | 이영희 | - | 2026-05-10 |

## 상태 범례

| 아이콘 | 상태 | 설명 |
|--------|------|------|
| 🟢 | 진행중 | 현재 개발이 진행 중인 스펙 |
| 🟡 | 보류 | 일시적으로 중단된 스펙 (사유를 PROGRESS.md에 기록) |
| ✅ | 완료 | 개발 및 QA가 완료된 스펙 |
| ❌ | 폐기 | 기획 변경 등으로 폐기된 스펙 (삭제하지 않고 상태만 변경) |
```

### 4-3. spec.md — 기능 설계 문서

각 스펙 폴더 내의 `spec.md`는 해당 기능의 설계를 정의하는 문서입니다. AI가 구현할 때 가장 핵심적으로 참조하는 파일입니다.

**포함 항목**

| 항목 | 설명 | 필수 |
| --- | --- | --- |
| 개요 | 이 기능이 무엇이고 왜 필요한지 | O |
| 기술 스택 | 사용하는 기술, 라이브러리 등 | O |
| 실행 환경 | 실행에 필요한 환경 | O |
| 실행 스크립트 | 빌드 또는 테스트 실행 방법 | △ |
| 환경 변수 | 필요한 설정 값 (민감 정보 제외) | △ |
| 데이터 구조 | 핵심 타입, 데이터 모델 정의 | O |
| 파일(페이지) 구성 | 이 기능에서 생성/수정하는 파일 목록 | O |
| API 엔드포인트 | 서버 통신이 필요한 경우의 API 정의 | △ |
| 변경 이력 | 스펙 문서 자체의 수정 내역 | O |

**작성 예시 (웹앱)**

```markdown
# 사용자 인증

## 개요
이메일/비밀번호 기반 로그인 및 회원가입 기능.
JWT 토큰을 사용한 인증 유지, 소셜 로그인(Google, GitHub) 지원.

## 기술 스택
- Next.js 14 (App Router)
- NextAuth.js v5
- Prisma + PostgreSQL

## 실행 환경
- Node.js 20+
- PostgreSQL 15+

## 데이터 구조

### User
| 필드 | 타입 | 설명 |
|------|------|------|
| id | string | UUID |
| email | string | 이메일 주소 (unique) |
| name | string | 표시 이름 |
| role | UserRole | 사용자 역할 |
| createdAt | DateTime | 생성 시각 |

### UserRole
| 값 | 설명 |
|------|------|
| ADMIN | 관리자 |
| USER | 일반 사용자 |

## 파일 구성
| 파일 | 경로 | 설명 |
|------|------|------|
| auth.ts | src/lib/ | NextAuth 설정 |
| login/page.tsx | src/app/(auth)/ | 로그인 페이지 |
| register/page.tsx | src/app/(auth)/ | 회원가입 페이지 |
| useAuth.ts | src/hooks/ | 인증 상태 훅 |

## API 엔드포인트
| 메서드 | 경로 | 설명 |
|--------|------|------|
| POST | /api/auth/register | 회원가입 |
| POST | /api/auth/login | 로그인 |
| GET | /api/auth/me | 현재 사용자 정보 |

## 변경 이력
| 날짜 | 변경 내용 |
|------|----------|
| 2026-05-18 | 최초 작성 |
```

### 4-4. PROGRESS.md — 진행 상황 추적

각 스펙 폴더 내의 `PROGRESS.md`는 해당 기능의 개발 진행 상황을 추적하는 파일입니다.

**작성 예시**

```markdown
# 사용자 인증 — 진행 상황

## 영역별 진행도

| 영역 | 상태 | 비고 |
|------|------|------|
| DB 스키마 (Prisma) | ✅ 완료 | |
| NextAuth 설정 | ✅ 완료 | |
| 로그인 페이지 | 🔨 진행중 | 소셜 로그인 버튼 추가 중 |
| 회원가입 페이지 | ⬜ 미착수 | |
| API 라우트 | ⬜ 미착수 | |

## 작업 히스토리

### 2026-05-20
- 로그인 페이지 UI 구현
- Google OAuth 연동

### 2026-05-19
- Prisma User 모델 생성
- NextAuth 기본 설정

## 진행중
- [ ] GitHub OAuth 연동
- [ ] 로그인 폼 validation

## 다음 할일
- [ ] 회원가입 페이지 구현
- [ ] API 라우트 구현

## 결정 기록

### 인증 라이브러리 선택 (2026-05-19)
- **결정:** NextAuth.js v5
- **대안:** Clerk, Supabase Auth
- **사유:** 셀프 호스팅 가능, App Router 공식 지원, 소셜 로그인 통합 용이
```

## 5. 파일 간 역할 분리 기준

| 정보 | 작성 위치 | 이유 |
| --- | --- | --- |
| 프로젝트 전체 맥락 | AI 컨텍스트 파일 | AI가 가장 먼저 읽는 진입점 |
| 전체 스펙 목록과 상태 | INDEX.md | 한눈에 전체 현황 파악 |
| 기능의 설계 (무엇을, 어떤 구조로) | spec.md | 변하지 않는 설계 정보 |
| 기능의 진행 (어디까지, 무엇이 남았는지) | PROGRESS.md | 수시로 변하는 진행 정보 |
| 스펙 문서 자체의 수정 내역 | spec.md 변경 이력 | 설계 변경 추적 |
| 개발 중 기술적 결정과 사유 | PROGRESS.md 결정 기록 | 구현 중 발생한 의사결정 |

### 겹침을 방지하는 원칙

1. **설계 정보는 spec.md에만 존재한다.** 다른 파일에서 설계 정보가 필요하면 spec.md를 참조하도록 안내한다.
2. **진행 정보는 PROGRESS.md에만 존재한다.** AI 컨텍스트 파일이나 spec.md에 진행 상황을 적지 않는다.
3. **전체 현황은 INDEX.md에만 존재한다.** 개별 스펙의 상태를 AI 컨텍스트 파일에 나열하지 않는다.
4. **AI 컨텍스트 파일은 가볍게 유지한다.** 프로젝트 전체에 해당하는 정보만 작성하고, 나머지는 `.specs/`로 안내한다.

## 6. 스펙 주도 개발 워크플로우

### 6-1. 전체 흐름

| 순서 | 단계 | 행동 | 갱신 파일 |
| --- | --- | --- | --- |
| 1 | 스펙 생성 | `.specs/` 하위에 스펙 폴더 생성, spec.md 초안 작성 | spec.md, INDEX.md |
| 2 | 설계 작성 | 데이터 구조, 파일 구성, API 등 상세 설계 | spec.md |
| 3 | 개발 시작 | AI 도구에 spec.md를 맥락으로 전달하고 구현 시작 | PROGRESS.md |
| 4 | 진행 갱신 | 작업 완료 시마다 PROGRESS.md 갱신 | PROGRESS.md |
| 5 | 설계 변경 | 개발 중 설계가 변경되면 spec.md 갱신 | spec.md |
| 6 | 기능 완료 | 모든 영역 완료 시 INDEX.md 상태를 "완료"로 변경 | INDEX.md, PROGRESS.md |
| 7 | 아카이브 | 완료된 스펙은 삭제하지 않고 유지 | - |

### 6-2. 단계별 상세

**단계 1: 스펙 생성**

```bash
mkdir -p .specs/spec_user_auth
touch .specs/spec_user_auth/spec.md
touch .specs/spec_user_auth/PROGRESS.md
```

INDEX.md에 새 스펙을 추가합니다.

**단계 2: 설계 작성**

기획서, 회의 내용, 참고 자료를 바탕으로 spec.md를 작성합니다.

**단계 3~4: 개발 및 진행 갱신**

spec.md를 맥락으로 전달하고 구현을 시작합니다. 작업이 완료될 때마다 PROGRESS.md를 갱신합니다.

**단계 5: 설계 변경**

개발 중 데이터 구조나 설계가 변경되면 spec.md를 먼저 갱신하고 구현합니다.

**단계 6: 기능 완료**

모든 영역의 구현이 끝나면 INDEX.md의 상태를 갱신합니다.

## 7. 프로젝트에서의 활용

### 7-1. 멀티 스펙 병렬 개발

다수의 기능이 서로 다른 일정으로 개발될 때, 각 기능을 독립적인 스펙 폴더로 관리하여 병렬 개발을 지원합니다.

스펙 폴더는 Git 브랜치와 1:1로 매핑되는 것이 이상적입니다.

| --- | --- |
| spec_user_auth | feat/user_auth |
| spec_dashboard | feat/dashboard |
| spec_payment_flow | feat/payment_flow |

### 7-3. 스펙 간 의존성 관리

일부 기능은 다른 기능이 완료되어야 개발을 시작할 수 있습니다. 이런 의존성은 INDEX.md에서 관리합니다.

## 9. 스펙 작성 시 주의사항

### 9-1. 민감 정보 포함 금지

스펙 문서에는 API 키, 토큰, 비밀번호, 내부 서버 주소 등을 포함하지 않습니다. 스펙 문서는 Git에 커밋되며, 외부 AI 도구에 전달될 수 있습니다.

### 9-2. 스펙 범위를 적절히 설정하기

하나의 기능 브랜치로 개발하고 하나의 PR로 병합할 수 있는 단위가 적절합니다.

### 9-3. 완료된 스펙은 삭제하지 않는다

완료된 스펙 폴더는 삭제하지 않고 유지합니다. 이후 유지보수 시 참고 자료가 됩니다.

### 9-4. 스펙 문서를 코드와 함께 커밋한다

스펙 문서 변경은 관련 코드 변경과 같은 커밋 또는 같은 PR에 포함시킵니다.

### 9-5. AI가 스펙을 갱신할 때 확인한다

AI 도구가 PROGRESS.md를 자동 갱신하는 경우, 내용이 정확한지 반드시 확인합니다.
