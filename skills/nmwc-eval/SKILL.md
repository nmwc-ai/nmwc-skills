# NMWC 구현 검증 (nmwc-eval)

`nmwc-prd`·`nmwc-spec`이 정한 계약(AC·정책·설계)을 실제 코드와 실제 화면이 지키고 있는지 확인하는 스킬. 검증은 항상 두 갈래다 — **코드가 맞게 짜였는가**(§2)와 **실제로 켜서 써보면 맞는가**(§3). 코드만 읽고 "구현했다"고 믿는 것, 화면만 훑고 "괜찮아 보인다"고 믿는 것 둘 다 이 스킬이 막으려는 실패다.

이 스킬은 **검증 전용**이다. FAIL을 발견해도 코드를 고치지 않는다 — 결과를 EVAL.md에 남기고 구현자(별도 세션/에이전트)에게 넘긴다. 검증한 사람이 그 자리에서 자기 검증 결과를 고쳐버리면 검증의 의미가 없다.

## 0. 시작 전 — 검증 대상 확정

1. **기능 식별**: 사용자가 기능명·`.specs/spec_<feature>` 경로·PR을 지정하면 그걸 쓴다. 모호하면(`git status`/`git diff`로 최근 변경 파일을 보고) 어떤 기능을 검증할지 되묻는다 — 잘못된 대상을 검증하고 PASS를 주는 것보다 한 번 묻는 게 낫다.
2. **정본 확보**:
   - PRD: `docs/prd/<feature>/<feature>-PRD.md` — 있으면 필수 입력. AC·Non-Requirements·Principles의 출처.
   - Spec: `.specs/spec_<feature>/spec.md` + `PROGRESS.md` — 있으면 필수 입력. 설계(데이터 모델·파일 구성·기술 구현)의 출처.
   - **둘 다 없으면 이 스킬을 쓸 수 없다.** 대조할 정본이 없는데 "잘 구현됐다"는 판단은 지어내는 것이다. 사용자에게 먼저 `nmwc-prd`/`nmwc-spec`으로 정본을 만들라고 안내한다. 하나만 있으면(예: spec만 있고 PRD는 없음) 있는 것만으로 검증하고 EVAL.md에 "PRD 없음 — AC 기준 검증은 생략, spec 설계 대조만 수행"이라고 명시한다.
3. **구현 범위 확정**: `git diff <base>...HEAD` 또는 `git status`로 이 기능과 관련된 변경 파일을 추린다. spec.md의 "파일(페이지) 구성" 표에 있는 파일은 diff에 없어도 반드시 확인 대상에 포함한다(이미 존재하던 파일을 수정 안 하고 넘어갔을 수도 있다).

## 1. 검증 체크리스트 만들기

PRD와 spec.md를 읽고 세 갈래 체크리스트를 만든다. 이후 §2·§3에서 각 항목에 PASS/PARTIAL/FAIL을 매긴다.

1. **AC 체크리스트** — PRD의 각 `US-x.y` 아래 AC 표(`references/user-story-craft.md`와 같은 `| AC-n | 인수 조건 |` 형식)를 그대로 가져온다. **하나도 빠뜨리지 않는다** — AC 하나를 조용히 건너뛰는 것이 이 스킬이 가장 경계하는 실패다.
2. **정책 체크리스트** — PRD의 Non-Requirements(하지 않기로 한 것 — 구현에 몰래 들어와 있으면 FAIL) + Principles(지키기로 한 제약).
3. **설계 체크리스트** — spec.md의 데이터 모델·파일 구성·기술 구현·API가 실제 코드 구조와 일치하는가.

## 2. 코드 검증

체크리스트의 각 항목에 대해 실제 코드를 읽고 판정한다. **파일명만 보고 짐작하지 않는다** — 열어서 로직을 확인한다.

- **AC별 매핑**: AC 문장에서 동작·화면·조건을 키워드로 뽑아 grep → 구현 파일을 찾는다. 정상 경로 AC는 그 로직이 실제로 그렇게 동작하는지, 엣지·네거티브 케이스 AC(빈 값·중복·권한 없음·동시성 등)는 그 케이스를 실제로 처리하는 코드가 있는지 — "아마 되겠지"가 아니라 코드 라인으로 확인한다.
- **정책 대조**: Non-Requirements에 적힌 것이 실제로 구현에 없는지(스코프 크립 탐지), Principles가 요구하는 제약(예: "관리자 승인 없이 자동 발송 금지")이 코드 경로상 실제로 지켜지는지.
- **설계 대조**: spec.md의 데이터 모델(타입/필드)이 실제 타입 정의와 일치하는지, "파일(페이지) 구성" 표의 파일이 실제로 그 역할을 하는지.
- **회귀 확인**: 프로젝트에 테스트·타입체크·빌드 스크립트가 있으면 실행한다(`package.json`의 `test`/`typecheck`/`build` 등). 실패하면 그 자체로 관련 AC들을 FAIL 또는 PARTIAL로 내린다.
- 판정마다 근거를 `파일:라인` 형태로 남긴다. 근거 없는 PASS는 EVAL.md 검증 게이트(§6)에서 걸러진다.

## 3. 브라우저 검증 (claude-in-chrome) — 필수

코드가 맞아 보여도 실제로 켜서 눌러보면 다르게 동작하는 경우가 실무에서 가장 흔하다. **UI가 있는 기능이면 이 단계를 생략하지 않는다.** 절차와 판단 기준은 `references/browser-verification.md`를 읽고 따른다. 요지:

- `claude-in-chrome` 도구가 로드돼 있지 않으면 먼저 로드하고, 대상 기능의 로컬 개발 서버가 떠 있는지 확인한다(없으면 사용자에게 실행 방법을 묻거나 `run` 스킬을 참고한다).
- 각 User Story를 **실제 유저처럼** 처음부터 끝까지 밟는다 — "[역할]은 [행동]한다" 문장 그대로 재현하고, 그 과정에서 관련 AC(정상 + 엣지 케이스)를 하나씩 실제로 트리거해 확인한다. 코드를 읽고 유추하는 게 아니라 브라우저에서 직접 일으켜본다.
- **시각적 검증**을 병행한다 — 레이아웃 위계, 여백·정렬, 빈/로딩/에러 상태, AC의 "Prototype or Design"에 참조가 있으면 그것과 비교. 콘솔·네트워크 에러도 확인한다.
- 근거로 스크린샷을 남긴다 (§5 산출물 위치).
- **브라우저 검증을 생략했다면 그 사실과 사유를 EVAL.md에 명시한다.** 생략해놓고 PASS라고 쓰지 않는다 — 이건 이 스킬이 존재하는 이유 자체를 무너뜨리는 행동이다.

## 4. 판정 기준

| 판정 | 조건 |
|------|------|
| PASS | 관련 AC·정책·설계 항목 모두 코드+브라우저 양쪽에서 확인됨 |
| PARTIAL | 핵심 경로는 통과했지만 엣지 케이스·시각적 디테일이 미흡하거나, 브라우저 검증을 일부만 수행함 |
| FAIL | AC 미충족, Non-Requirements 위반, 설계와 불일치, 또는 회귀(테스트/빌드 실패) 발견 |

**전체 판정**은 하나라도 FAIL이 있으면 FAIL, FAIL 없이 PARTIAL이 있으면 PARTIAL, 전부 PASS면 PASS.

## 5. 산출물 — EVAL.md

`references/eval-template.md`의 구조를 그대로 따라 `.specs/spec_<feature>/EVAL.md`에 쓴다(spec.md=설계·PROGRESS.md=진행과 같은 자리 — 검증 결과의 단일 출처). 스크린샷은 `.specs/spec_<feature>/eval-screenshots/`에 저장하고 EVAL.md에서 상대 경로로 참조한다. `.specs/` 구조가 없는 프로젝트라면(스펙 없이 PRD만 있는 경우) `docs/prd/<feature>/EVAL.md`에 쓴다.

## 6. 검증 게이트 (필수 — 통과 전 완료 금지)

EVAL.md를 다 쓰면 완료를 선언하기 전에 반드시 검증기를 통과시킨다.

```bash
python3 scripts/check_eval.py .specs/spec_<feature>/EVAL.md docs/prd/<feature>/<feature>-PRD.md
```

이 검증기는:
- PRD의 모든 AC-ID가 EVAL.md에 빠짐없이 등장하는지(조용히 건너뛴 AC 적발)
- FAIL·PARTIAL 판정에 근거(빈 칸이 아닌 실제 내용)가 있는지
- 전체 판정 필드가 있는지
- 브라우저 검증 섹션이 비어 있지 않은지(생략했다면 사유가 적혀 있는지)

를 프로그램으로 검사한다. **exit 0(PASS)가 나올 때까지 완료하지 않는다.**

## 7. 마무리

- 전체 판정이 **FAIL**이면: 완료 선언을 하지 않는다. FAIL 항목과 근거를 요약해 사용자에게 보고하고, 구현을 고치는 것은 별도 패스(사용자 또는 다른 에이전트)로 넘긴다. 고친 뒤에는 이 스킬을 다시 돌려 재검증한다 — "고쳤으니 이번엔 될 것"이라는 선언만으로 끝내지 않는다.
- 전체 판정이 **PASS**면: EVAL.md·스크린샷 위치를 안내하고 완료를 선언한다.
- 전체 판정이 **PARTIAL**이면: 무엇이 부족한지 구체적으로 짚고, 사용자에게 지금 완료로 볼지 보완할지 확인한다.
