# 브라우저 검증 절차 (claude-in-chrome)

코드 리뷰는 "이렇게 동작하도록 짜여 있다"만 말해준다. 실제로 그렇게 동작하는지는 켜서 눌러봐야 안다 — 이 문서는 그 절차다.

## 0. 도구 로드

`claude-in-chrome` 도구가 아직 로드되지 않았다면(도구 목록에 스키마가 안 잡혀 있으면) 필요한 도구를 **한 번에** 로드한다. 하나씩 로드하지 않는다 — 매번 왕복이 든다.

```
ToolSearch: "select:mcp__claude-in-chrome__tabs_context_mcp,mcp__claude-in-chrome__navigate,mcp__claude-in-chrome__computer,mcp__claude-in-chrome__read_page,mcp__claude-in-chrome__tabs_create_mcp,mcp__claude-in-chrome__read_console_messages,mcp__claude-in-chrome__read_network_requests,mcp__claude-in-chrome__find,mcp__claude-in-chrome__get_page_text"
```

## 1. 실행 환경 확인

1. spec.md의 "실행 환경"·"라우팅" 섹션에서 로컬 개발 서버 URL과 대상 경로(`/<path>`)를 확인한다.
2. 서버가 떠 있는지 확인한다(`curl -sI <url>` 또는 이미 알려진 정보로). 안 떠 있으면:
   - 프로젝트에 `run` 스킬이나 명확한 실행 커맨드가 있으면 그걸로 띄운다.
   - 애매하면 사용자에게 "개발 서버를 실행해달라"고 요청하고, 실행되면 이어간다. **서버 없이 코드만 보고 브라우저 검증을 통과시킨 척하지 않는다.**
3. `tabs_context_mcp`로 현재 탭 상태를 확인한 뒤, 이 검증 전용으로 새 탭을 연다(`tabs_create_mcp`) — 사용자가 쓰던 탭을 재사용하지 않는다.

## 2. User Story별 플로우 재현

PRD의 각 `US-x.y` 문장("[역할]은 [행동]한다")을 실제로 그 역할이 된 것처럼 처음부터 끝까지 밟는다.

- **정상 경로**: 문장 그대로 수행하며, 관련 AC가 하나씩 실제로 참이 되는지 확인한다(예: "로그인하면 대시보드로 들어간다" → 로그인 폼 채우고 제출 → URL이 실제로 `/dashboard`로 바뀌는지 `read_page`/`get_page_text`로 확인).
- **엣지·네거티브 케이스**: AC에 적힌 실패·경계 조건을 실제로 일으켜본다(빈 값 제출, 잘못된 값, 권한 없는 상태로 접근, 같은 동작 중복 실행 등). "아마 막아줄 것"이 아니라 실제로 막히는지 확인한다.
- 한 플로우가 막히면(요소를 못 찾음, 예상과 다른 화면) 2-3회 다른 방식으로 시도해보고, 그래도 안 되면 **거기서 멈추고** 해당 AC를 FAIL 또는 "확인 불가"로 기록한다. 같은 실패를 무한히 반복하지 않는다.

## 3. 대화상자(alert/confirm/prompt) 회피

JavaScript `alert`/`confirm`/`prompt`나 브라우저 네이티브 모달을 트리거하는 액션(예: "삭제" 버튼)은 피한다. 이후 모든 브라우저 명령이 멈춘다. 꼭 지나가야 하는 플로우면 먼저 콘솔에서 관련 로직을 확인하는 것으로 대체하거나, 사용자에게 알리고 진행 여부를 확인한다.

**한글 등 조합 문자(IME) 입력 주의**: 자동화 타이핑이 한글 입력을 실제 필드 값으로 못 옮기는 경우가 있다(조합 이벤트가 제대로 전달되지 않아 입력창이 비어 보임). 입력값 자체가 검증 대상이 아니면 영문/숫자 테스트 값으로 대체하고, 타이핑 직후 스크린샷이나 `read_page`로 실제 입력된 값을 확인한 뒤 다음 단계로 넘어간다 — 입력이 비어 있는데 "입력했다"고 가정하고 진행하면 엉뚱한 AC가 FAIL로 잘못 찍힌다.

## 4. 시각적 검증

각 주요 화면에서:

- **레이아웃 위계**: 중요한 정보가 실제로 눈에 먼저 들어오는 위치에 있는가. 여백·정렬이 깨진 곳은 없는가.
- **상태별 확인**: 빈 상태(데이터 없음)·로딩 상태·에러 상태가 각각 의도대로 보이는가 — 정상 경로만 보고 넘어가지 않는다.
- **Prototype or Design 참조**: PRD의 AC 표에 프로토타입/디자인 링크가 있으면 그것과 나란히 비교해 눈에 띄게 다른 점을 기록한다. 링크가 없으면("없음") 이 항목은 생략한다 — 없는 기준을 지어내 비교하지 않는다.
- 반응형이 spec.md에 언급돼 있으면(모바일/데스크톱) 뷰포트를 바꿔가며 확인한다.

## 5. 숨은 에러 확인

각 플로우 후 `read_console_messages`(에러/경고 패턴)와 `read_network_requests`(4xx/5xx, 실패한 요청)를 확인한다. 화면은 멀쩡해 보여도 콘솔에 에러가 쌓이는 경우가 흔하다 — 발견하면 근거로 남기고 관련 AC 판정에 반영한다.

## 6. 증거 남기기

주요 단계(성공 상태, 그리고 실패를 재현했다면 그 실패 상태)마다 스크린샷을 찍어 `.specs/spec_<feature>/eval-screenshots/<설명적-파일명>.png`로 저장한다. 파일명은 `ac-2-no-redirect.png`처럼 어떤 AC/상황인지 알 수 있게 짓는다 — 나중에 EVAL.md만 보고도 무엇의 증거인지 알 수 있어야 한다.

## 7. 막히거나 생략했을 때

- 도구가 연결되지 않았거나, 개발 서버를 띄울 수 없거나, 반복 시도에도 플로우가 막히면 — 억지로 우회하지 않는다. EVAL.md의 "브라우저 검증" 섹션에 **무엇을 왜 못 했는지** 명시하고, 관련 AC는 "브라우저 검증 생략 — 코드 검증만으로 판정"으로 표시한다. 안 한 걸 한 것처럼 쓰지 않는다.
