# 폴리싱 모드 (1-4단계)

요청 대상이 **기존 파일 경로**일 때 이 모드로 진행한다.

## 1. Discover

작업 착수 전 `validate_code`(또는 대상 프로젝트의 lint CLI, 예: `nmwc-design lint <path>`)를 먼저 돌려 baseline 위반 수를 확보한다. 이 숫자가 이후 진행 상황 보고의 기준이 된다.

## 2. Reference

위반 목록에서 다루는 값(raw hex, 임의 px, 레거시 alias 등)마다 필요한 토큰 카테고리를 `get_token`으로 확인해 대응 범위를 파악한다.

## 3. Apply — violation 순회

lint 위반을 하나씩 순회한다:
- raw hex/px 값 → `suggest_token`으로 가장 가까운 토큰을 받아 치환.
- 레거시 alias(`--bg`, `--card` 등) → 프로젝트 CLAUDE.md에 정식 토큰 매핑이 있으면 그대로 정식 이름으로 치환, 없으면 `get_token`으로 동의어를 찾는다.
- 인라인 `:root` 오버라이드 → 삭제하고 오버라이드하려던 값이 이미 spec에 있는 토큰인지 확인. 없으면 사용자에게 새 토큰이 필요한지 질문(임의로 추가하지 않는다).

## 4. Verify — 검증 게이트

`validate_code`/lint를 **참고가 아니라 게이트**로 취급한다.

1. 수정 후 lint 재실행 (검증 1회차).
2. violation 0 → 완료 보고 (baseline 대비 몇 건 수정했는지 명시).
3. violation > 0 → 남은 위반에 대해 2-3단계를 반복한 뒤 재검증. **재시도는 최대 3회** — 즉 검증 1회차(위 1번) + 재시도 3회 = 총 최대 4회 lint 실행. 카운트는 파일 단위로 누적한다(위반 항목 개별 카운트가 아니다).
4. 재시도 3회를 모두 소진한 뒤에도 위반이 남아 있으면 자동 수정을 멈추고, 남은 위반 목록(파일·라인·이유)을 사용자에게 명시적으로 보고한다. "완료"라고 보고하지 않는다.

이 3회 상한은 무한루프 방지용이다. 3회 안에 안 풀리는 위반은 대개 도구의 알려진 한계(예: JSX inline `style={{}}` 객체의 dimension 값은 raw hex만 검사되고 radius/spacing은 미검사)이거나 사람 판단이 필요한 케이스다.
