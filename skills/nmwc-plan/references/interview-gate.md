# 인터뷰 게이트 — 모호성 제거 상세 가이드

> 목적: 문서를 쓰기 전에 **진짜 의도**와 **필수 확정 항목**을 파악한다.
> 표면 요청("이 기능 만들어줘")이 아니라, 그 뒤의 동기와 성공 기준을 캔다.
> 질문 수 제한 없음 — 충분히 이해될 때까지. 단, 사용자가 지치지 않게 한 번에 하나씩.

---

## 0. 언제 얼마나 인터뷰하나

요청을 두 축으로 빠르게 가늠한다.

|  | **모호** (목표·범위·대상 중 2개+ 불명확) | **구체적** (실행 가능 수준) |
|---|---|---|
| **새 기획** (PRD/제품 결정 필요) | **A. 풀 인터뷰** | **B. 경량 확인 1~2개** |
| **구현 설계** (Spec, 이미 뭘 만들지 앎) | **C. 시나리오·범위만 캐기** | **D. 바로 진행** |

- 이미 PRD·요구사항 문서가 있고 그 연장선이면 게이트를 건너뛰고 바로 라우팅한다 (`> 기존 요구사항 기반으로 진행합니다`).
- 사용자가 "바로/급해/알아서" 신호를 주면 즉시 중단하고 진행하되, 빈칸은 `[확인 필요]`로 남긴다.

---

## 1. 필수 확정 항목 (체크리스트)

아래가 채워져야 좋은 PRD/Spec이 나온다. 각 항목이 흐릿하면 어떤 슬롭이 생기는지도 함께 적는다 — 이걸 알면 무엇을 캐야 하는지 분명해진다.

### 1-1. 의도·배경 (왜 지금)
- 표면: "뭘 만드시려고요?" → 심층: "이걸로 어떤 변화를 만들고 싶으세요? 왜 지금이 그 시점인가요?"
- 안 캐면: "사용자 경험을 혁신한다" 같은 배경. → PRD의 Background가 공허해진다.

### 1-2. 성공 지표 (Goal)
- "무엇이 얼마나 개선되면 성공인가요? 그걸 어떻게 측정하죠?"
- 측정 지표가 없으면 "지표 없음(정성 목표)"임을 명시적으로 확인한다 — 추측으로 숫자를 지어내지 않는다.
- 안 캐면: 나중에 성공 여부를 못 따지는 목표. → Goal 슬롭.

### 1-3. 대상·역할 (누가)
- "누가 이걸 쓰나요? 역할이 나뉘나요(예: 소유자 vs 대행자, 관리자 vs 일반)? 역할마다 할 수 있는 게 다른가요?"
- 안 캐면: 주어 없는 유저 스토리, 권한 경계 누락. → User Story·AC 슬롭.

### 1-4. 범위 경계 (할 것 / 안 할 것 / 나중)
- "이번에 반드시 되어야 하는 건 뭔가요? 이번엔 **안** 할 건 뭔가요? 나중으로 미룰 건?"
- "이것만은 절대 포함/제외해야 하는 게 있나요?" (숨은 제약)
- 안 캐면: 무한 확장, Non-Requirements 공백. → Scope 슬롭.

### 1-5. 핵심 시나리오 (실제 행동)
- "사용자가 이걸 열면 첫 번째로 뭘 하나요? 그다음은요? 잘 풀리면 어떤 상태가 되죠?"
- "잘 안 풀리는 경우(실패·에러·경계)는 어떤 게 있나요?" — **엣지/네거티브 케이스는 여기서 캔다.**
- 안 캐면: 추상적 기능 나열, Epic·Story로 분해 불가. → User Story 뼈대 없음.

### 1-6. 제약 (기술·일정·정책)
- "지켜야 할 기술 스택·플랫폼이 있나요? 마감은? 법/정책상 걸리는 게 있나요?"
- 안 캐면: 구현 불가능한 AC, 뒤집히는 설계.

## 2. 질문 기법

| 기법 | 언제 | 예시 |
|------|------|------|
| **래더링** | 표면 답 → 깊은 동기 | "그게 왜 중요한가요?" → "그게 되면 뭐가 달라지죠?" |
| **5 Whys** | 근본 원인 | "왜 그렇게 됐나요?" 반복 |
| **Jobs-to-be-Done** | 진짜 해결할 문제 | "이 작업이 끝나면 어떤 상황이어야 하나요?" |
| **시나리오** | 구체적 상상 유도 | "실제로 화면을 연다고 상상하면, 뭘 먼저 누르나요?" |
| **대비 질문** | 방향·범위 명확화 | "A 방식과 B 방식 중 어디에 가깝나요?" → `AskUserQuestion`으로 |
| **반사 질문** | 확인 + 심화 | "말씀하신 건 [요약]이라는 뜻인가요? 더 있나요?" |
| **엣지 프로빙** | 네거티브 케이스 발굴 | "그게 실패하면요? 두 명이 동시에 하면요? 값이 비면요?" |

**개방형은 자유서술, 선택형은 AskUserQuestion.** "왜/의도/시나리오"는 사람이 풀어서 말해야 하므로 한 개씩 텍스트로 묻는다. "A vs B / 포함할까 말까 / 어느 범위까지" 같은 이산 선택은 `AskUserQuestion`으로 2~4개 보기를 제시한다 (권장안을 첫 보기 + "(권장)"). 한 번에 최대 몇 개까지만.

## 3. 진행 절차

**1단계 — 인지 + 첫 질문(의도 중심)**
```
이해했습니다. [요청 요약] 작업이군요.
제대로 도와드리고 싶어서 하나 여쭤볼게요.
[의도/배경을 캐는 첫 질문 — "왜" 중심]
```

**2단계 — 한 개씩 심층 인터뷰**
이전 답변을 반영해 §1 체크리스트의 빈칸을 채운다. 같은 정보가 반복되거나, 접근을 자신 있게 제안할 수 있으면 종료.

**3단계 — 인사이트 정리 + 접근 제안**
내부 분류 근거는 노출하지 않는다. 사용자에게는 깔끔하게:
```
인터뷰로 파악한 핵심:
- 진짜 의도: [...]
- 핵심 제약/성공 기준: [...]
- 범위: [할 것 / 안 할 것]

이걸 바탕으로 제안드립니다:
**방식 1 (권장)** — 이유 / 장단점
**방식 2** — 이유 / 장단점
어떤 방향이 맞으실까요? 조합도 가능합니다.
```

**4단계 — 라우팅**
승인된 방향 + 확정 항목을 요약해 `nmwc-prd` 또는 `nmwc-spec`로 넘긴다. 하위 스킬이 같은 걸 다시 묻지 않도록, 확정된 것과 `[확인 필요]`로 남은 것을 명확히 구분해 전달한다.

---

## 4. 하위 스킬로 넘기는 컨텍스트 형식

```markdown
## 인터뷰 확정 요약
- 의도·배경: ...
- 성공 지표: ... (또는 "정성 목표 — 지표 없음")
- 대상·역할: ...
- 범위 — 할 것: ... / 안 할 것: ... / 나중: ...
- 핵심 시나리오: 1) ... 2) ...
- 엣지·네거티브 케이스: ...
- 제약: ...
- 승인된 접근: ...

## 확인 필요 (추측 금지, 문서에 그대로 남길 것)
- [확인 필요: ...]
```
