# AI Slop 제거 원칙 — 상세 + 예시

> "AI slop"은 그럴듯하지만 알맹이가 없는 문장이다. 팀 협업에서 슬롭은 리뷰를 겉돌게 하고,
> 구현자가 무엇을 만들어야 할지 모르게 만든다. 아래 원칙은 PRD·Spec 양쪽에 적용된다.
> 기준선(gold standard)은 "커머스 예시 — User Story"의 AC들이다: 전부 관찰 가능하고, 엣지·네거티브 케이스를 담는다.

---

## 원칙 1 — 추측 금지, `[확인 필요]` 사용

모르는 값을 그럴듯하게 지어내는 것이 가장 흔한 슬롭이다. 인터뷰에서 못 채운 항목은 그 자리에 남긴다.

```
❌ (지어냄)  목표: MAU를 30% 늘린다.
✅ (정직)    목표: [확인 필요: 어떤 지표를 얼마나 개선하면 성공인지]
```

`[확인 필요: ...]`는 부끄러운 게 아니라 문서의 정직함이다. 리뷰어가 채울 지점을 정확히 가리킨다.

## 원칙 2 — 모든 문장은 검증 가능하게 (특히 AC)

AC는 읽는 사람이 **pass/fail을 판정**할 수 있어야 한다. "잘 처리된다"가 아니라 "무엇이 어떻게 되는지".

```
❌  결제가 잘 처리된다.
✅  결제를 마치고 "구매 완료"를 누르면 주문이 배송준비중으로 이동한다.

❌  에러를 우아하게 처리한다.
✅  등록 실패 시 마켓 원문 오류 대신 "원인 요약 + 해결 단계" 형식의 안내가 표시되고,
    필요하면 원문 메시지를 펼쳐 볼 수 있다.
```

**엣지·네거티브 케이스를 포함한다.** 좋은 AC 묶음은 정상 경로만이 아니라 실패·경계·중복·동시성을 담는다.
```
✅  송장번호가 비어 있으면 발송 처리 버튼이 눌리지 않는다.
✅  이미 구매 처리된 주문에는 구매 완료를 중복 실행할 수 없다.
✅  같은 링크를 다시 시도해도 실패 안내가 중복으로 쌓이지 않는다.
```

## 원칙 3 — 형용사 슬롭 금지

공허한 수식어를 관찰 가능한 행동·수치로 바꾼다.

| 슬롭 단어 | 대신 |
|-----------|------|
| 혁신적인 / 강력한 / 최적화된 | 무엇을 얼마나 하는지 수치·행동으로 |
| 원활한 / seamless / 매끄러운 | 어떤 단계가 사라지거나 몇 초 안에 되는지 |
| 직관적인 / 사용하기 쉬운 | 어떤 조작으로 무엇이 되는지 |
| 다양한 / 여러 가지 | 정확히 몇 개, 무엇무엇 |
| robust / 안정적인 | 어떤 실패에 어떻게 대응하는지 |

```
❌  강력한 검색으로 원하는 상품을 다양한 방식으로 빠르게 찾는다.
✅  주문번호·주문자·전화번호·상품명으로 검색할 수 있다.
```

## 원칙 4 — 범위 밖을 명시한다

무엇을 **안** 하는지 적어야 리뷰가 실질적이 된다. Non-Requirements / Future Work / "이번엔 제외"를 비워두지 않는다.

```
✅  결제 자체는 서비스가 대신하지 않는다(정책상 사람이 직접 수행).
✅  Non-Requirements: 자동 발주 확인 (재고 확인 전 접수 방지 위해 항상 수동)
```

## 원칙 5 — 구체 > 추상

숫자·구체 명사·실제 화면 용어를 쓴다. 추상 명사 나열을 피한다.
```
❌  다양한 알림을 적절한 채널로 보낸다.
✅  알림 채널 3개(배포·시스템 상태 / 장애·에러 / 주문)를 등록·수정할 수 있다.
```

## 원칙 6 — 한국어 존댓말, 팀 공용 용어

문서 본문은 팀이 실제로 쓰는 말로 쓴다. 번역투("~하는 것을 가능하게 한다")를 피하고, 도메인 용어(셀러·리스팅·발주·송장 등)를 그대로 쓴다.

- [ ] 모든 AC가 pass/fail 판정 가능한가?
- [ ] 엣지·네거티브 케이스가 있는가? (정상 경로만 있으면 부족)
- [ ] 형용사 슬롭(혁신적·원활한·다양한…)이 남아 있지 않은가?
- [ ] 안 하는 것(범위 밖)이 적혀 있는가?
