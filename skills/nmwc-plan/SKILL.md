# NMWC 기획 문서 시스템 — 진입점 (nmwc-plan)

NMWC 팀원 누구든 **같은 규격으로 기획**하고, 협업 과정에서 **AI slop(모호한 미사여구·검증 불가능한 문장·추측으로 메운 빈칸)** 없이 소통하도록 만드는 기획 문서 하네스의 진입점이다.

이 스킬은 직접 문서를 쓰지 않는다. 세 가지 일을 한다.

1. **분류** — 지금 필요한 게 PRD인지, Spec인지, 둘 다인지 판별한다.
2. **인터뷰 게이트** — 문서를 쓰기 전에 모호성을 제거한다. 이게 이 시스템의 핵심이다.
3. **라우팅** — 정제된 요구사항을 들고 `nmwc-prd` / `nmwc-spec`로 넘긴다.

> **왜 인터뷰가 먼저인가.** 슬롭은 대개 글쓰기 실력이 아니라 **불명확한 입력**에서 나온다. 목표·대상·범위·성공 기준이 흐릿한 채로 쓰면, 모델은 빈칸을 그럴듯한 미사여구로 메운다. 인터뷰로 이 빈칸들을 먼저 채우면, 뒤따르는 PRD·Spec은 구체적이고 검증 가능한 문장으로 채워진다. 인터뷰는 형식이 아니라 slop 방지 장치다.

## 1. 두 산출물의 관계

| 산출물 | 질문 | 담당 스킬 | 무엇을 확정하나 |
|--------|------|-----------|-----------------|
| **PRD** | "왜 · 무엇을 · 누구를 위해" | `nmwc-prd` | 배경·목표·가설·범위, 그리고 **Epic → User Story → 인수 조건(AC)** |
| **Spec** | "어떻게 만들 것인가" | `nmwc-spec` | 데이터 모델·화면 구성·기술 구현. `.specs/` 폴더에 기능 단위로 |

PRD 하나의 Requirements는 **여러 개의 Spec**으로 쪼개진다 (1 Spec = 1 기능 브랜치 = 1 PR). PRD는 제품 층위, Spec은 기능 층위다.

## 2. 실행 흐름 (유연)

이 시스템은 세 가지 진입을 모두 지원한다. 사용자의 요청 문맥으로 판별하되, 애매하면 **한 번만** 물어본다.

- **새 제품·기능의 기획을 처음 시작** → 인터뷰 → PRD → (원하면) "이어서 Spec도 만들까요?"로 연속 진행
- **PRD는 이미 있고 특정 기능을 구현 설계** → 인터뷰(가벼움) → Spec 바로 작성. 관련 PRD가 있으면 참조하되, 없어도 진행한다
- **기존 기능을 스펙 문서화** (PRD 없이) → 인터뷰(가벼움) → Spec

```
요청 접수
   │
   ├─ 분류: PRD? Spec? 둘 다? 불확실? ──(불확실하면 1회 질문)
   │
   ▼
인터뷰 게이트 (references/interview-gate.md) ── 모호성 제거
   │
   ▼
라우팅
   ├─ PRD 필요 →  nmwc-prd 실행
   │                 └─ 완료 후: "이 PRD의 기능을 Spec으로 이어서 설계할까요?"
   └─ Spec 필요 →  nmwc-spec 실행
```

## 3. 인터뷰 게이트 (모호성 제거)

문서를 쓰기 전, 아래 **필수 확정 항목**이 채워질 때까지 인터뷰한다. 상세 기법·질문 예시는 `references/interview-gate.md`를 읽고 따른다.

| 확정 항목 | 이게 흐릿하면 생기는 슬롭 |
|-----------|--------------------------|
| **의도·배경 (왜 지금)** | "사용자 경험을 혁신한다" 같은 공허한 배경 |
| **성공 지표 (Goal)** | 측정 불가능한 목표, 나중에 성공 여부를 못 따짐 |
| **대상·역할 (누가)** | 주어 없는 유저 스토리, 권한 경계 누락 |
| **범위 경계 (할 것/안 할 것/나중)** | 무한히 부풀거나, 뭘 안 하는지 안 적혀 리뷰가 겉돔 |
| **핵심 시나리오 (실제 행동)** | 추상적인 기능 나열, Epic·Story로 못 쪼갬 |
| **제약 (기술·일정·정책)** | 구현 불가능한 AC, 뒤늦게 뒤집히는 설계 |

**인터뷰 진행 원칙** (상세는 reference):
- **한 번에 하나의 초점 질문**. 개방형(왜·의도·시나리오)은 자유서술로 묻고, 선택형(A안 vs B안, 범위 포함 여부)은 `AskUserQuestion`으로 묻는다.
- 이전 답변을 반영해 다음 질문을 도출한다 (적극적 경청).
- 당연해 보여도 가정은 명시적으로 확인한다 — "혹시 ~라고 가정하고 계신가요?"
- **탈출**: 사용자가 "바로 해줘 / 급해 / 알아서" 라고 하면 즉시 인터뷰를 멈추고 진행한다. 단, 못 채운 항목은 문서에 `[확인 필요: ...]`로 **남겨두고 절대 추측으로 메우지 않는다**.
- **종료**: 필수 항목이 채워지고 접근 방식을 자신 있게 제안할 수 있으면, 인사이트 3줄 요약 + 접근 제안(2~3안) → 승인받고 문서 작성으로 넘어간다.

## 4. AI slop 제거 원칙 (모든 산출물 공통)

이 원칙은 PRD·Spec 양쪽에 그대로 적용된다. 상세·예시는 `references/anti-slop.md`.

1. **추측 금지, `[확인 필요]` 사용.** 모르는 값은 그럴듯하게 지어내지 않는다. 인터뷰에서 못 채웠으면 그 자리에 `[확인 필요: 질문]`을 남긴다.
2. **모든 문장은 검증 가능하게.** 특히 AC는 읽는 사람이 pass/fail을 판정할 수 있어야 한다.
3. **형용사 슬롭 금지.** "혁신적·원활한·다양한·강력한·최적화된·직관적인·seamless·robust" 류 공허한 수식어를 관찰 가능한 행동·수치로 바꾼다.
4. **범위 밖을 명시한다.** 안 하는 것을 적어야 리뷰가 실질적이 된다.
5. **한국어 존댓말, 팀 공용 용어.** 문서 본문은 팀이 실제로 쓰는 말로.

## 5. 라우팅 실행

정제가 끝나면 해당 스킬을 실행한다.
- **PRD**: `nmwc-prd` 스킬을 실행한다 (설치돼 있으면 `/nmwc-prd`, 아니면 같은 마켓플레이스의 `skills/nmwc-prd/SKILL.md`를 읽어 따른다). 인터뷰에서 확정한 필수 항목을 컨텍스트로 넘긴다.
- **Spec**: `nmwc-spec` 스킬을 실행한다 (`/nmwc-spec` 또는 `skills/nmwc-spec/SKILL.md`).
- **둘 다**: PRD를 먼저 완성한 뒤, PRD의 Requirements를 기준으로 Spec으로 이어간다.

넘길 때는 인터뷰로 확정한 내용을 요약해서 전달하고, 하위 스킬이 같은 걸 다시 묻지 않게 한다.
