# Epic · User Story · 인수 조건(AC) 작성 규율

> 이 부분이 PRD의 심장이다. 기준선은 "커머스 예시 — User Story"다.
> 목표: 구현자가 무엇을 만들어야 할지, QA가 무엇을 확인해야 할지 문서만 보고 알 수 있게.

---

## 1. Epic

- 관련된 기능을 묶은 큰 단위. **번호 + 제목**으로 표기 (`1. 계정·팀·로그인`, `2. 상품 수집`).
- Scope의 **Requirements에서 도출**한다. Requirement에 해당하지 않는 Epic은 만들지 않는다.
- 제목은 사용자 관점의 명사구. 시스템 내부 구조("DB 스키마")가 아니라 사용자가 하는 일("상품 수집").
- 커머스 레퍼런스의 7개 에픽: 계정·팀·로그인 / 상품 수집 / 판매 준비(리스팅) / 마켓 등록·관리 / 주문 처리 / 운영 알림 / 기본 설정. — 사용자 여정 순서로 배열돼 있다.

## 2. User Story

**ID**: `US-{에픽번호}.{스토리번호}` — `US-1.1`, `US-1.2`, `US-2.1` ...

**제목**: 짧은 명사구 (`팀원 초대`, `주문 자동 수신`).

**한 문장 스토리**: `[역할]은 [행동]한다` 형태. 역할 주어를 반드시 넣는다.
```
US-1.1 팀 계정으로 로그인·로그아웃
  → "셀러는 이메일과 비밀번호로 로그인하고, 끝나면 로그아웃한다."

US-5.1 주문 자동 수신
  → "셀러는 마켓에 들어온 주문을 서비스에서 자동으로 받아 본다."
```
- **역할(누구)**을 명확히. 역할이 나뉘면(소유자/대행자 등) 스토리마다 어떤 역할인지 드러낸다.
- 한 스토리 = 하나의 사용자 목표. 두 개가 섞이면 스토리를 나눈다.

## 3. 인수 조건(AC) — 가장 중요

각 스토리 아래에 검증 가능한 조건들을 나열한다. **AC가 곧 완료의 정의(Definition of Done)** 이자 QA 체크리스트다.

### 3-1. AC 품질 규칙

1. **pass/fail 판정 가능.** 읽는 사람이 "됐다/안 됐다"를 명확히 가를 수 있어야 한다.
   - ❌ "결제가 잘 처리된다" → ✅ "결제를 마치고 '구매 완료'를 누르면 주문이 배송준비중으로 이동한다."
2. **관찰 가능한 결과.** 내부 구현이 아니라 사용자가 보는 것 기준.
   - ❌ "trie 자료구조로 검색을 인덱싱한다" → ✅ "주문번호·주문자·전화번호·상품명으로 검색할 수 있다."
3. **정상 경로 + 엣지·네거티브 케이스.** 좋은 AC 묶음은 실패·경계·중복·동시성·빈 값을 담는다. 이게 슬롭과 진짜 스펙을 가른다.
   - "송장번호가 비어 있으면 발송 처리 버튼이 눌리지 않는다."
   - "이미 구매 처리된 주문에는 구매 완료를 중복 실행할 수 없다."
   - "취소·반품된 주문에는 송장을 등록할 수 없다."
   - "같은 링크를 다시 시도해도 실패 안내가 중복으로 쌓이지 않는다."
4. **모순 방지 조건도 AC.** 상태 간 일관성을 명시한다.
   - "상태는 '연결됨'인데 수집만 계속 실패하는 모순이 없어야 한다."
5. **강제/비강제를 구분.** 경고인지 차단인지 분명히.
   - "경고는 등록을 강제로 막지 않지만(최종 판단은 셀러), 등록 직전 점검 단계에서 한 번 더 보인다."
6. **한 AC = 한 조건.** 여러 개가 접속사로 이어지면 쪼갠다 (단, 자연스러운 인과 "~하면 ~된다"는 하나로 둔다).

### 3-2. AC 개수 감각
- 스토리당 2~7개. 단순하면 2개, 복잡하면 7개. 커머스 레퍼런스 평균 ~4개.
- 정상 경로만 3개보다, 정상 2 + 엣지 2가 낫다.

### 3-3. 정본 .md의 AC 표 형식 (NMWC 표준)
```markdown
#### US-1.1 팀 계정으로 로그인·로그아웃
> 셀러는 이메일과 비밀번호로 로그인하고, 끝나면 로그아웃한다.

| ID | 인수 조건 |
| --- | --- |
| AC-1 | 이메일과 비밀번호로 로그인하면 대시보드로 들어간다. |
| AC-2 | 로그인하지 않고 주소로 바로 들어가려 하면 로그인 화면으로 보내지고, 로그인 후 원래 가려던 화면으로 이어진다. |
| AC-3 | 이메일·비밀번호가 틀리면 안내가 보인다(어느 쪽이 틀렸는지는 알려주지 않음). |
| AC-4 | 비밀번호는 화면에 다시 노출되지 않는다. |
| Prototype or Design | 없음 |
| Additional Details | 없음 |
```

## 4. JSON 스키마 (렌더 입력)

정본 .md를 쓴 뒤, 같은 내용을 아래 JSON으로 구조화해 `<feature>-user-story.json`에 저장한다.
`render_user_story.py`가 이걸 읽어 HTML 뷰를 만든다. 통계(에픽/스토리/AC 수)는 스크립트가 자동 집계한다.

```json
{
  "title": "커머스 예시 — User Story",
  "subtitle": "구매대행 자동화 서비스의 사용자 스토리와 인수 조건",
  "footer": "NMWC · User Story",
  "epics": [
    {
      "id": 1,
      "title": "계정·팀·로그인",
      "stories": [
        {
          "id": "US-1.1",
          "title": "팀 계정으로 로그인·로그아웃",
          "sentence": "셀러는 이메일과 비밀번호로 로그인하고, 끝나면 로그아웃한다.",
          "criteria": [
            "이메일과 비밀번호로 로그인하면 대시보드로 들어간다.",
            "로그인하지 않고 주소로 바로 들어가려 하면 로그인 화면으로 보내지고, 로그인 후 원래 가려던 화면으로 이어진다.",
            "이메일·비밀번호가 틀리면 안내가 보인다(어느 쪽이 틀렸는지는 알려주지 않음).",
            "비밀번호는 화면에 다시 노출되지 않는다."
          ]
        }
      ]
    }
  ]
}
```

필드:
- `title`, `subtitle`, `footer` — 문서 헤더/푸터 (subtitle·footer는 선택).
- `epics[]`: `id`(정수), `title`(문자열), `stories[]`.
- `stories[]`: `id`(문자열 `US-x.y`), `title`, `sentence`(한 문장 스토리), `criteria`(AC 문자열 배열).
- 정본 .md의 `Prototype or Design` / `Additional Details`는 HTML 뷰에는 넣지 않는다 (뷰는 스토리+AC 중심). 필요하면 `story.details`(문자열, 선택)로 추가 가능.
